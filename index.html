What's different

Personas, not throwaway sessions.

Most AI tools treat every conversation as a blank slate. BawtHub treats each bot as an entity that persists across turns, surfaces, and even tasks dispatched to other bots.

Everywhere else

A new agent every time
  • ✕ "New chat" resets context. Markdowns and system prompts are all you carry forward.
  • ✕ Agents are single sessions. No history, no identity, no relationship.
  • ✕ Each tool lives in its own silo. Web chat, CLI, voice — three different products.
  • ✕ You pick a model per app. Locked to whatever that vendor ships.
  • ✕ Agents can't coordinate with each other. You're the messenger.

BawtHub

A roster you keep coming back to
  • ✓ Bots are persistent entities. Loopy remembers what you talked about last week. Snark knows your codebase.
  • ✓ Segregated memory per bot, with a shared cross-bot history pool. They can pull each other's threads.
  • ✓ Web, voice, CLI all hit the same session. Type in the browser, continue in the terminal.
  • ✓ Any model behind one OpenAI-compatible API. Local GGUF, OpenAI, Grok, Claude, Ollama, vLLM — swap per bot.
  • ✓ Bots message each other. Loopy asks Vex to scan a host. Snark hands a refactor to Caid. You watch it happen.
How it's wired

One API. Every model. Every surface.

llm-bawt is the OpenAI-compatible spine. BawtHub is the web/voice frontend. Agent backends plug in over a Redis-backed bridge with streaming tool events. Everything funnels into the same memory layer.

Agent task system

Plan, dispatch, review.

Projects are markdown definitions of a component. Tasks are work items under them. An agent refines the plan, dispatches it to a bot with a clean context, and the result lands in a dashboard waiting for sign-off.

Origin

It started as a CLI.

BawtHub didn't start as a product. It grew, piece by piece, from a tiny shell command to a multi-bot platform with voice, memory, and an agent task system. Same core API the whole way.

v0 · CLI

One command, one bot.

llm -b loopy "what's up" — local, file-based message history. The whole thing was a CLI for chatting with LLMs. The idea of named bots with their own prompts was there from day one.

v1 · llm-bawt server

One API, every model.

Server-side backend. Same bots, reachable from any machine. Every provider — OpenAI, Grok, Claude, local GGUF, Ollama, vLLM — merged behind a single OpenAI-compatible API. Memory moved into Postgres + pgvector.

v2 · unmute → BawtHub

Voice. Then a real UI.

Started as unmute — a full-stack realtime voice chat framework. Low-latency STT, TTS, hotword detection. Grew the web UI around it; voice never went away.

v3 · agent bridges

Agents in the same chat.

Started with OpenClaw. Then Claude Code SDK over the same bridge. Then Codex. Plus native tool-loop on the non-agent bots. The hard problem: streaming tool calls + results from every backend into one beautiful UI.

v4 · task system

Projects, tasks, sign-off.

Plan a project in markdown. Drop a task. Agent refines it. Bot executes in clean context. You review. The whole pipeline is in the dashboard — and the agents themselves can call MCP tools to manage tasks, pull history, and ping each other.
