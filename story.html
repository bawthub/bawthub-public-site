How you end up watching the agents.

BawtHub grew, piece by piece, from a one-line shell command into a self-hosted platform where a team of coding agents builds software — and you steer by watching them work. This is how it got there.

The through-line is one idea that kept getting sharper: don't fire a message into a black box. See what the agent is doing while it does it — the tool calls and their results, rendered live. Everything below is the road to that.

v0 · CLI roots

The whole stack starts at llm-bawt, which began life as a CLI for messaging LLMs:

llm "what's up"
llm -b loopy "what's up"  # talk to a specific bot

Originally local and file-based message history. Eventually moved server-side so the same bots could be reached from any machine. The idea of named bots with their own prompts was baked in from the very first commit.

v1 · unified API

llm-bawt merges every LLM provider behind a single OpenAI-compatible API: Anthropic Claude (via the Agent SDK), OpenAI, Grok (xAI), local GGUF models, Ollama, vLLM. Each bot has its own system prompt, model preference, voice, and tool access.

The memory layer landed on Postgres + pgvector, with multiple types:

• Message history — raw conversation turns
• Semantic memories — facts that decay unless reinforced; contradictions resolved
• Summaries — rolled-up context for long threads
• User profile — attributes with confidence scores and provenance
• Per-bot segregated memory + a shared cross-bot history pool

v2 · unmute → BawtHub

Started as unmute — a full-stack realtime voice chat framework. Low-latency STT, TTS, hotword detection. The web UI grew off that and became the home, putting more weight on text chat, but voice has always stayed a first-class feature. The 3D avatar with VRM/GLB lip-sync ships in the same UI. (The frontend is now a Vite + React 19 SPA — it migrated off Next.js.)

v3 · agent bridges

Started with OpenClaw, fitting external agents into the same chat UI as regular bots — and evolving the concept that would become the whole point: realtime streaming of tool-call request and response. Built a Redis backend and a per-agent-type bridge.

When Claude OAuth was banned from OpenClaw, the Claude Code SDK integration landed over the same bridge pattern. Then Codex joined. Plus native tool-call/loop on the non-agent bots. Every backend, one common event protocol.

Streaming those events is the focal point — and the biggest engineering struggle in the whole project. Tool calls and their responses, from every agent backend, rendered beautifully in one UI, streamed into a Zustand client store so flipping between bots is instant and threads keep going.

v3.5 · cross-surface continuity

Web, voice, and CLI all hit the same core API. You can have a chat in the browser and continue the same thread from the command line — same session, same memory, same bot. A real feature — just not, it turns out, the headline.

v4 · agent task system

Projects are high-level components, each with its own markdown definition. Tasks are work items inside them. The lifecycle:

1. Create — title, description, optional image attachments
2. Plan — an agent refines the task into a detailed markdown plan
3. Dispatch — the plan goes to a bot in a clean context with only the project markdown + the task markdown
4. Execute — bot works, streams progress via REST callbacks
5. Review — bot marks the task ready for review with a summary
6. Approve / revise — you sign off, or send it back; the agent revises

Agents set tasks to review — only a human marks them done. They have MCP tools for managing tasks/projects and tapping the full memory layer. The same memory the chat bots use is available to agents working on tasks.

v5 · observable agentic development

This is where it all points. Everything before was plumbing; this is the thesis. The Relay: a live timeline of every agent's every turn. Turn cards carry exact time, latency, token/context pressure, and model. +/− diff chips on every Write and Edit. Sub-agent tool calls nested under their parent Agent card. Questions and approvals streamed in as they happen.

The bet underneath it: most agentic tools hide the work — a spinner, a "thinking…", a collapsed log, a clean final diff. BawtHub does the opposite. The failure mode of agentic coding isn't "it can't code"; it's that it confidently did the wrong thing three steps deep and you didn't notice. Making the work visible is the fix. It's the part that took hundreds of hours, and it's why the app is still private.

Why this is different.

It's tempting to say "persistent personas" — and that's real. The usual agent is a single session: you spawn one, it runs, and when the session ends it's gone. BawtHub treats each bot as a constant entity you go back to by name — with prompts, memories, voices, and roles you define and edit over time.

But the deeper difference is the center of gravity. No IDE renders the work this way, because IDEs are built around a human typing with the agent bolted on as a sidecar. BawtHub is built around the agent working, with visibility as the primary surface. It's a remote, always-on dev box with a family of agents at the keyboard — and you watching the whole board.

"I can be working with Snark on something, then go over to Loopy or Vex and have them doing their own thing — but have them message each other to coordinate. The shared history lets one bot do something and another just pull the thread and continue. And the whole time, you can see exactly what each one is doing."

Bots talk to each other via mcp__bots_send_message. Loopy pings Vex when something needs a security touch. Snark hands a refactor to Caid. Gavel makes the ship-or-hold call on a diff. Cross-bot history is shared, so a task started by one bot can be finished by another — and you steer all of it by watching, not by hoping.
