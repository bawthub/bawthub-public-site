Every tool call.

The chat surface is the dense, visible output of everything underneath. Every backend — the native tool loop, Claude Code, Codex, OpenClaw — emits tool events into a single stream, and the chat renders them through one shared component tree. Diffs. Parsed commands. SSH targets. MCP pills. Streaming partials. All shaped to be skimmable, foldable, and beautiful.

01 What's actually hard about this.

Most chat UIs render strings. Tool-call chat UIs render structured events arriving out of order, sometimes interleaved, sometimes recovered after a reload. Each backend emits its own event shape — Claude Code's tool_use blocks, Codex's item-typed updates, OpenClaw's WS frames, the native loop's streamed deltas. The chat surface normalizes all of them into one event model, then renders.

The interesting work is in the renderer. Tool calls are first-class: not text dumps inside a bubble, but their own composable component with status, kind, command, output, and a chevron that folds the body. Bash gets a terminal-themed block. Edit gets a unified diff. SSH gets a parsed host pill and an inner command. MCP gets the magenta pill the agents use. Errors flip the status dot red and surface the exit code in the headline so the most-asked questions never need a click.

02 The unified event model.

Every backend reports tool activity through the same event shape. The chat surface doesn't know whether a Bash event came from the native tool loop or from a Claude Code subprocess — it only knows that a tool with kind bash, status ok, an args blob, and a result blob just landed.

The unified shape is defined as AgentEvent in llm_bawt/agent_backends/openclaw/events.py (named for historical reasons — OpenClaw was the first bridge, and the shape stuck when Claude Code and Codex were retrofitted). The frontend imports the same shape via the generated Prisma + tRPC types. Renderers dispatch on event.kind; new tool kinds add a renderer without changing anything upstream.

03 Tool-call rendering — the design vocabulary.

Every tool kind gets its own component, but they all share the same skeleton: status dot, kind pill, headline, size, chevron, and a body that's hidden by default for long output. The kind pill colour and the body styling are all that change between renderers.

KindPillHeadline showsBody
Bash Bash Bolded binary + dimmed args; pipes collapsed Black terminal-themed block; $ prompt + output
Edit Edit File path + size delta (+8 −3) Unified diff with add/rm/context line markers and the real hunk header
Read Read File path + line count Mono body with head-of-file slice; long files get an ellipsis-and-tail
Grep / Search Search Pattern + scope path + match count Match list with file:line prefixes
SSH SSH Host pill + → + the command actually run there Whatever the inner command produced (often terminal)
MCP MCP Magenta pill with the namespaced tool name + dimmed args Structured result; bolds, spans, and the tool's native shape

The status dot in the head has three states with distinct colours: ✓ ok, ● running (pulses), ! error (with the exit code shoved into the size slot). Errors never collapse — they default open.

04 Parsed commands.

Every bash and ssh call gets parsed at render time, not just dumped as a string. The renderer pulls out:

  • Host for SSH calls — rendered as a coloured pill (echo, unraid-lan) so the target is the first thing you see
  • Command name (the binary) — bolded
  • Args — dimmed mono
  • Pipes — collapsed into one line if the chain is short, expanded inline if long
  • Heredocs — folded into the body, not the headline

This is the difference between reading "this command ran" and understanding what the command did at a glance. The parser lives at frontend/src/lib/commandParser.ts and is used by both the Bash and SSH renderers.

05 The MCP pill.

Tools that come in via the MCP server get a distinctive rendering — a magenta pill (⌬ memory_search) with the full namespaced tool name, the args inline and dimmed, and the result body collapsible. This is how bots use cross-cutting capabilities like memory, task management, or messaging other bots. The pill makes "this is an MCP tool, not a local command" instantly obvious in a long thread.

The renderer for kind: "mcp" doesn't try to be smart about the body — it shows the JSON-ish result the server returned, formatted as monospace, and lets the human's eye do the parsing. Bots themselves are usually the consumers; the human just wants to scan whether the lookup worked.

06 Grouping & collapse.

Some turns produce a dozen tool calls in a row — searches, file reads, edits. Showing them all expanded would bury the conversation. The chat applies a heuristic: if a single message has more than 3 sequential tool calls of similar kinds, they collapse under a summary header.

The grouped view shows the count and a one-line summary the renderer composes from the kinds — "Read 7 files, ran 4 searches, executed 1 grep" — so the human can decide whether to expand. Inside the group, each tool call is collapsed by default. Click any individual chevron to drill in.

07 Streaming, not batching.

Tool calls render as they happen. A bash command in flight shows a pulsing yellow dot and the placeholder streaming… in the size slot. When the stream completes, the dot flips to green, the size resolves to the actual byte count, and the body is fully populated.

The two-spout problem and how we got around it.

The chat-completions SSE stream is tied to a single HTTP request — it dies on disconnect, so a page reload mid-tool-call loses everything in flight. The bridges also publish events to a per-(bot, user) unified stream (events:{bot_id}:{user_id}) that the frontend keeps a persistent EventSource against. Tool events are written to both: the SSE for ordering and the unified stream for resilience. The frontend dedupes on event id. See the streaming events page for the full mechanics.

Streaming behavior also drives the renderer's incremental mode: rather than waiting for a tool's result and then mounting the whole component, the renderer mounts the tool head as soon as the tool_use_started event lands, then progressively appends to its body as tool_use_delta events arrive. For long bash output or large file reads, you see the output filling in real time, terminal-style.

08 Why this matters.

Most chat surfaces with tool support treat tool calls as noise — they collapse them, dim them, gray them out. They're treated as boilerplate between the human and the model.

BawtHub treats them as the point. When a bot does work for you, the doing is what you want to see — what it ran, what came back, what changed. The chat surface is engineered as much for tool readability as for prose readability. Both are first-class.

"How agent tool calls are mapped into a common rendering engine for doing edits, seeing the changes with diffs, parsed commandlines and ssh."

This page is that. Every detail above — kinds, statuses, parsed hosts, diffs, the MCP pill, grouping, streaming dots — is a direct artifact of treating "the agent did something useful" as a UX problem worth solving end-to-end, not as a developer convenience.

frontend/src/components/chat/ChatUI.tsx
The root chat surface. ~3,500 lines. Owns the Zustand store wiring, message list, event subscription, and dispatches to per-kind tool renderers.
frontend/src/components/chat/ToolCallCard.tsx
The base tool-call component. Renders header (status + kind pill + parsed command + size + chevron) and body (kind-specific renderer).
frontend/src/components/chat/renderers/
Per-kind renderers. BashRenderer.tsx · EditRenderer.tsx (the diff view) · MCPRenderer.tsx · SSHRenderer.tsx · ReadRenderer.tsx · etc.
frontend/src/lib/commandParser.ts
Bash/SSH command parsing. Extracts host, binary, pipes, heredocs from raw command strings for richer rendering.
frontend/src/stores/chatStore.ts
Zustand store. Holds messages keyed by session, tool-call events keyed by id, dedupes between SSE and unified-stream sources.
frontend/src/lib/eventSource.ts
Persistent EventSource client. Subscribes to events:{bot_id}:{user_id}; reconnects with backoff; per-window consumer id to avoid Redis consumer-group races.
src/llm_bawt/agent_backends/openclaw/events.py
The unified event shape. AgentEvent dataclass — kind, status, name, args, result, tokens, ts. The contract every bridge implements.
